# Domain Corpus for Term Priority Scoring

**Goal:** Score keyword term-worthiness using inverse frequency against a **service/user-manual domain corpus**, not general English — so domain-generic parts ("screw", "bracket") rank lower than product-specific or branded terms.

**Context:** Extraction stays exhaustive. Priority is computed post-aggregation from `Keyword` callout frequency + `Concept` embedding cohesion + corpus rarity.

---

## 1. Overall Goal & Scope

**What:** Build a static domain frequency table from service-manual PDF text, then score each project keyword:

```
domainSpecificity = log(1 + docFreq) × log(1 + corpusDocCount / corpusTermDocFreq)
consistencyROI    = docFreq × (1 - meanCohesion)
tbxPriority       = w1 × domainSpecificity + w2 × consistencyROI
```

Where:
- `docFreq` = callout/occurrence count for the keyword in the current project
- `corpusTermDocFreq` = number of manuals in the corpus that contain the term (document frequency)
- `meanCohesion` = mean cosine similarity of concept embeddings to centroid (existing `similarityService`)

**Why domain corpus beats general English:**
- "screw" is rare in general English relative to "the", but **common across service manuals** → low domain specificity → deprioritize
- "biodrill retention clip" may be absent from general dictionaries but **also rare in the manual corpus** → high specificity if frequent in *this* doc
- Matches BM25/TF-IDF intuition: high TF in document × high IDF across domain corpus

**Out of scope (v1):** UI tiers, auto-filtering extraction, multilingual corpora.

---

## 2. Discussion: Challenges & Solutions

### Corpus source

| Approach | Pros | Cons |
|----------|------|------|
| **Bundled open manuals** | Offline, reproducible, no API | Need openly licensed PDFs; corpus may not match user's industry |
| **Build script on user PDF folder** | Corpus matches user's domain | User must supply extra manuals; cold start |
| **Cross-project Termcaller DB** | Grows with usage | Empty at hackathon demo; privacy if multi-tenant |

**Recommendation:** PR 1 ships a **build script + committed frequency JSON** derived from open service/installation manuals (see `test_data/`). PR 2 optionally aggregates Termcaller projects.

### Licensing

Do not commit proprietary manuals. Commit only derived frequency counts, or a script users run locally. Open sources: Archive.org appliance manuals, manufacturer install guides marked for download, iFixit-style repair docs where license permits text mining.

### Multi-word terms

Tokenize on whitespace; for phrases like `retention clip`, score on **minimum** domain frequency of content words, or treat normalized full phrase as a key if it appears as a unit in corpus.

### Cold start (single manual)

With one document, `docFreq` still works; corpus IDF distinguishes domain-generic vs bespoke. Cohesion requires ≥2 concepts; flag `insufficientEvidence` when `docFreq < 3`.

## 3. PR Implementation Plan

### PR 1 — Corpus build pipeline + static data

**Goal:** Produce `backend/data/service-manual-corpus-freq.json`.

**Scope:**
1. `scripts/buildManualCorpus.ts` — scan PDF folder via `PdfSession`, extract page text, tokenize (reuse `normalizeSourceTerm` rules per token).
2. Record per term: `termFrequency`, `documentFrequency` (manuals containing term), `zipf` optional.
3. Seed corpus from 3–5 open manuals in `test_data/manuals/` (user adds PDFs; `.gitignore` if needed).
4. Vitest: known generic ("screw") has higher `documentFrequency` than a synthetic rare token.

### PR 2 — `termPriorityService`

**Goal:** API to rank keywords for a project.

**Scope:**
1. `corpusLookup.ts` — load JSON, `getDomainFrequency(term)`.
2. `termPriorityService.ts` — for each keyword: callout count, cohesion from embeddings, combine scores, assign tier A–D.
3. `GET /api/projects/:id/keywords/priority` — sorted list with score breakdown.

### PR 3 — UI surfacing

**Goal:** Keywords tab shows priority badge / sort-by-priority default.
