# TBX-Basic glossary export

**Goal:** Export a Termcaller project's concepts and keywords as a TBX-Basic `.tbx` file, interoperable with CAT tools and aligned with translate-bot's managed termbase export.

**Priority:** HIGH — UI already exposes a disabled "Export TBX" button in `DashboardHeader`.

**Status:** IMPLEMENTED (PR1 + PR2)

## 2. Termcaller data model → TBX mapping

| Termcaller | TBX-Basic |
|------------|-----------|
| `Concept` (one per unique definition) | `<termEntry id="c{conceptId}">` |
| `Concept.candidateConceptName` + `Concept.definitionText` | `<descrip type="definition">` on `termEntry` (concept-level) |
| `Keyword.sourceTerm` (grouped under concept) | `<tig><term>…</term></tig>` under one `<langSet xml:lang="en">` |
| — | `<termNote type="administrativeStatus">preferredTerm-admn-sts</termNote>` on each tig (single-language MVP) |
| `Project.name` | `<title>` in `martifHeader` |

Termcaller has no `subjectField`, multilingual terms, or `AdministrativeStatus` per keyword yet. MVP export: **one `langSet` per concept** (default `en`), one `tig` per keyword linked to that concept, concept definition on `termEntry`.

Keywords not linked to any concept: either omit or export as separate entries with term-only tigs (decide in PR).

## 3. XSLT in Node — evaluation

### Can we use XSLT?

Technically yes (`saxon-js`, `libxslt` / `xslt4node`), but **XSLT is the wrong tool for this job**:

- XSLT transforms **XML → XML**. Our source is **SQLite/Prisma**, not an intermediate XML document.
- translate-bot's proven path is **imperative DOM / builder**, not stylesheet-based generation.
- Adding XSLT would mean inventing an intermediate XML format solely to run a transform — extra complexity with no benefit.

### Recommended Node approach

**Programmatic XML build** (port of `TermbaseExportService`):

1. Query `Concept` + related `Keyword[]` for the project.
2. Build TBX-Basic structure with **`xmlbuilder2`** (or equivalent) — handles escaping, indentation, attributes.
3. Stream or buffer the result; respond with `Content-Type: application/xml` and `Content-Disposition: attachment; filename="{project}-termbase.tbx"`.

**Validation (tests / optional runtime):**

- translate-bot uses **Jing** (Java) against `TBXBasicRNGV02.rng`.
- Node options: shell out to `jing` in CI/vitest (same as Java), or defer strict RNG validation to manual/CI and assert structure in unit tests against `sample-basic.tbx` shape.
- Do **not** emit `<!DOCTYPE martif SYSTEM "coreStructure.dtd">` — breaks validators and DOM parsers (see translate-bot export tests).

### Alternatives considered

| Approach | Verdict |
|----------|---------|
| XSLT (`saxon-js`) | Overkill; no upstream reference for TBX |
| Template strings | Fragile for escaped text in definitions |
| `fast-xml-parser` `XMLBuilder` | Viable; `xmlbuilder2` is more ergonomic for nested TBX |
| Reuse Java microservice | Out of scope for hackathon monolith |
