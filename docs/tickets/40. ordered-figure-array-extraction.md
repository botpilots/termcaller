# Ordered figure array extraction

**Goal:** Stop asking the LLM to assign `figureNumber` on every extracted part. Return an ordered `figures[]` array (one entry per illustration in reading order); code assigns page-local numbers `1`, `2`, `3` from array index. Empty figure containers are ignored.

**Context:** Replaces per-concept `figureNumber` in the Gemini extraction schema (ticket 16). Downstream `ExtractionResult.extractedConcepts` stays flat for processing/DB.

---

## 1. Overall Goal & Scope

- Extraction LLM returns `{ figures: [{ parts: [...] }] }` instead of flat `extractedConcepts` with `figureNumber`.
- `normalizeExtractionResult` flattens to `ExtractedCallout[]`, skipping empty figures and assigning `figureNumber = index + 1` after filtering.
- `discoverAndValidate` validation uses the same ordered `figures[]` pattern for multi-figure page discovery.

---

## 2. Discussion: Challenges & Solutions

- **Reading order:** Still required from the LLM, but expressed as array order rather than repeated number tags — fewer consistency errors.
- **Empty figures:** Discouraged in prompt; if returned, dropped during normalization without affecting numbering of remaining figures.
- **Backward compatibility:** Internal `ExtractionResult` unchanged; only the Gemini response schema changes.
